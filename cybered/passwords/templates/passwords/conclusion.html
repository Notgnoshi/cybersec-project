Conclusion

We've covered a lot of different ideas related to passwords and password security, so let's go over the highlights.

Identification and Verification

There are two steps for authenticating a user to use a service.

  1. The user claims to have an identity on the service.
  2. The user verifies that they are, in fact, the owner of their claimed identity.

How are Passwords Stored?

We should always assume that attackers will eventually gain access to a service's password database. In order to limit any potential damage, services do not store the passwords. Instead, they store the hashes of the passwords.

Why Store Hashes?

Services store the password hashes because hash functions are one-way functions. This means that it's (almost) impossible to figure out the password from the hash.

Why Salt the Passwords?

Except that it is possible to find some passwords from their hashes using rainbow tables. So we add random stuff to each password before we salt them — that way they definitely will not be in any list of common passwords!

How to Crack Passwords

There are many approaches to cracking passwords if you have the password database — which contains the username, salt, and salted hash — but we only showed you two ways.

  1. Start at the beginning of a dictionary and start guessing words.
  2. Use a big list of commonly used passwords.

The second approach works the best for most passwords because people aren't very creative when they pick their passwords, and computers are really good at checking the password hashes thousands of times per second.

What Makes Passwords Strong?

Strong passwords are passwords that take a long time to crack. We used an XKCD comic to explain that the best way to make passwords strong is to make them long.
