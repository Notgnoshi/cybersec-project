Introduction

When a user is attempting to access some service, for example a website, their computer, or a mobile application, they often must gain authorization before using that service. Authentication is performed in two steps. First, the user must identify themselves. Second, they must verify that they are, in fact, the user they claim to be.

Allowing a user to log in only once they have provided the correct password is one extremely common method for performing this verification, but there are many others. In this module we will discuss how passwords are used, what makes some passwords stronger than other passwords, and how attackers might attempt to crack a password.

Module Goals

  • Understand how a password used to verify a user's identity
  • Demonstrate several strategies for cracking passwords
  • Examine how to prevent each strategy

Passwords are hashed as a damage control measure. People commonly use the same password for multiple services. If (when) a database is compromised, we want to limit the attacker's access to other services used by the same user.

In addition, the service webmaster or database administrator should not be able to view a user's passwords. Storing the hash makes it impossible to find a user's password.

However, just storing the hash is not enough. We must salt the password before hashing for several reasons. First, it renders lookup tables much more difficult to use. Second, if (when) an attacker gains access to a password database, they will not be able to tell if two different users happen to use the same password. Further, storing the salted hashes makes it impossible to simply transmit the hash of a password to another service used by the same user with the same password.
